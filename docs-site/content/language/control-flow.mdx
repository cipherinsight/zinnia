# Control Flow

Zinnia supports standard Python control flow constructs. Because programs compile to
static arithmetic circuits, every control flow construct must have a statically
determinable bound on its execution. The compiler uses SMT-based analysis to automatically
infer these bounds in most cases.

## If / Elif / Else

Conditional statements use standard Python syntax.

```python
if x > 0:
    result = x
elif x == 0:
    result = 0
else:
    result = -x
```

### Circuit Compilation: Guarded Joins

In an arithmetic circuit, both branches of an `if`/`else` are fully executed. The
compiler evaluates each branch symbolically and then merges the results using a
**guarded join**. The guard (the boolean condition) selects which branch's values
propagate.

Concretely, for a statement like:

```python
if cond:
    x = a
else:
    x = b
```

The circuit computes both `a` and `b`, then produces:

```
x = cond * a + (1 - cond) * b
```

This means both branches contribute constraints to the circuit, regardless of the
runtime value of `cond`. This is essential for zero-knowledge: the circuit structure
does not reveal which branch was taken.

For `elif` chains, the compiler nests the guarded joins:

```python
if c1:
    x = v1
elif c2:
    x = v2
else:
    x = v3
```

This compiles as:

```
x = c1 * v1 + (1 - c1) * (c2 * v2 + (1 - c2) * v3)
```

## While Loops

While loops repeat a block as long as a condition holds.

```python
i = 0
total = 0
while i < 10:
    total = total + i
    i = i + 1
```

### Inductive Unrolling

The compiler unrolls while loops by inductively analyzing the loop to determine a
static upper bound on the number of iterations. It uses an SMT solver to discharge
the boundedness proof.

#### Implicit Bounded Loops

In many cases, no explicit bound annotation is needed. The compiler infers bounds from
the loop condition and the update pattern:

```python
# The compiler infers that this loop runs at most n iterations
i = 0
while i < n:
    i = i + 1
```

If the compiler cannot automatically determine a bound, it will report a compilation
error asking you to restructure the loop or provide more information.

#### Loop Limit

The compiler enforces a configurable maximum iteration count. The default limit is
**1000 iterations**. If the proven bound exceeds this limit, compilation fails. You can
adjust this limit in the compiler configuration.

```python
# This is fine: compiler proves the bound is 100
i = 0
while i < 100:
    process(i)
    i = i + 1

# This will fail if max_loop_limit < 10000
i = 0
while i < 10000:
    process(i)
    i = i + 1
```

### Symbolic Execution of Loops

Each iteration of a while loop is symbolically executed. The loop body produces
constraints for every unrolled iteration, with path conditions ensuring that iterations
beyond the actual runtime count are effectively no-ops (their constraints are guarded
by the negation of the loop condition).

## For Loops

For loops iterate over `range(...)` or other iterables.

### Range-Based For Loops

```python
# Fixed range
total = 0
for i in range(10):
    total = total + i

# Range with start and stop
for i in range(1, 11):
    process(i)

# Range with step
for i in range(0, 20, 2):
    process(i)
```

Range-based for loops have a statically known iteration count (determined from the
range arguments) and are straightforward for the compiler to unroll.

### Iterable For Loops

```python
values = [10, 20, 30, 40]
total = 0
for v in values:
    total = total + v
```

When iterating over a list or array, the compiler determines the iteration count from
the length of the iterable.

## Break and Continue

`break` and `continue` are supported inside both `while` and `for` loops.

```python
# Find the first negative value
result = -1
for i in range(len(data)):
    if data[i] < 0:
        result = i
        break
```

```python
# Sum only positive values
total = 0
for v in values:
    if v <= 0:
        continue
    total = total + v
```

### Circuit Compilation

In the circuit, `break` and `continue` do not actually skip computation. Instead, they
refine the **path condition** for subsequent statements in the loop body. After a `break`,
the path condition for all remaining iterations becomes false, effectively nullifying
their contributions. After a `continue`, the path condition for the rest of the current
iteration becomes false.

This means the circuit always contains constraints for the full unrolled loop, but
guarded so that only the logically executed iterations affect the output.

## Early Return

Functions may return early from within conditional blocks or loops.

```python
def find_index(arr: NDArray[Int, 100], target: Integer) -> Integer:
    for i in range(100):
        if arr[i] == target:
            return i
    return -1
```

### Circuit Compilation

Early returns are handled via refined path conditions, similar to `break`. Once a
`return` is encountered along a path, the path condition for all subsequent statements
on that path becomes false. The final return value is the guarded merge of all possible
return points.

## Recursion

Zinnia supports recursive functions, provided the compiler can statically prove that
the recursion depth is bounded.

```python
def factorial(n: Integer) -> Integer:
    if n <= 1:
        return 1
    return n * factorial(n - 1)
```

```python
def fibonacci(n: Integer) -> Integer:
    if n <= 0:
        return 0
    if n == 1:
        return 1
    return fibonacci(n - 1) + fibonacci(n - 2)
```

### Bounded Recursion

The compiler uses SMT analysis to prove that each recursive call decreases a well-founded
measure (such as an integer argument) toward a base case. If the compiler cannot
establish a static bound, compilation fails.

#### Recursion Limit

The default recursion depth limit is **100**. This is configurable in the compiler
settings. If the proven recursion depth exceeds this limit, compilation fails.

### Circuit Compilation

Recursive calls are inlined up to the proven bound. Each level of recursion adds its
constraints to the circuit, guarded by the path condition that determines whether the
recursive case applies.

## Assert Statements

Assert statements introduce constraints into the circuit.

```python
assert x >= 0
assert x + y == expected
assert balance >= withdrawal, "insufficient funds"
```

### Circuit Compilation

An assert compiles to a circuit constraint that enforces the condition. The constraint
is guarded by the current path condition, so an assert inside an `if` block only applies
when that branch is taken.

```python
if mode == 1:
    assert x > 0    # only enforced when mode == 1
else:
    assert x <= 0   # only enforced when mode != 1
```

If the assert message string is provided, it is used for error reporting during
witness generation but does not affect the circuit.

## Pass Statements

`pass` is a no-op that produces no constraints.

```python
if condition:
    process(x)
else:
    pass  # do nothing in this branch
```
