# Non-Linear Functions

Arithmetic circuits support addition and multiplication natively, but transcendental
functions like `exp`, `log`, and `sin` have no direct field representation. Zinnia
approximates these functions using truncated Taylor series expansions, with the polynomial
degree chosen to guarantee that the approximation error falls below the fixed-point
precision threshold.

## Approximation Strategy

Each non-linear function f(x) is approximated by a polynomial of degree d:

```
f(x) approx sum(a_i * x^i, i = 0 to d) + R_d(x)
```

where `a_i` are the Taylor coefficients and `R_d(x)` is the Lagrange remainder term. The
compiler selects the smallest degree d such that the remainder is bounded below the
fixed-point precision:

```
|R_d(x)| < 2^(-s) = 2^(-64)
```

Since the machine epsilon is 2^(-64), any error smaller than this is invisible in the
fixed-point representation. The approximated value and the true value will map to the same
(or adjacent) field elements.

### Degree Selection

The degree d depends on the function and the input range. The Lagrange remainder for a
Taylor expansion about x = 0 is:

```
R_d(x) = f^(d+1)(c) * x^(d+1) / (d+1)!
```

for some c between 0 and x. The compiler computes a conservative upper bound on this
remainder over the valid input range and increases d until the bound drops below 2^(-64).

For functions with limited convergence radius, the compiler applies range reduction
techniques first. For example, `exp(x)` for large x is decomposed as
`exp(n + r) = exp(n) * exp(r)` where n is an integer and `|r| < 1`, so the Taylor series
for `exp(r)` converges rapidly.

### Circuit Cost

A degree-d polynomial requires:
- **d multiplications** (computing x^2, x^3, ..., x^d via successive multiplication)
- **d additions** (accumulating each a\_i * x^i term)

This makes the total circuit cost **2d constraints** (approximately), plus any constraints
for range reduction. The cost is independent of the precision -- the 64-bit fixed-point
precision determines d, and d determines the circuit cost.

## Function Details

### Exponential: exp(x)

The Taylor series for exp(x) about x = 0 is:

```
exp(x) = 1 + x + x^2/2! + x^3/3! + ... + x^d/d!
```

For large |x|, range reduction decomposes x = n + r where n = floor(x) and |r| &lt; 1. The
compiler precomputes exp(n) for the range of valid integer parts and evaluates exp(r) via
the Taylor series. The final result is `exp(n) * exp(r)`.

```python
x: Float = 2.5
y = math.exp(x)  # y approx 12.1825
```

### Natural Logarithm: ln(1 + x)

The Taylor series for ln(1 + x) about x = 0 is:

```
ln(1 + x) = x - x^2/2 + x^3/3 - x^4/4 + ...
```

This series converges for |x| &lt; 1. For `math.log(x)` with arbitrary positive x, the
compiler rewrites it as `ln(1 + (x - 1))` after range reduction to bring the argument
close to 1.

```python
x: Float = 0.5
y = math.log(1.0 + x)  # y approx 0.4055
```

### Trigonometric Functions: sin(x), cos(x), tan(x)

Sine and cosine use the standard Taylor expansions:

```
sin(x) = x - x^3/3! + x^5/5! - ...
cos(x) = 1 - x^2/2! + x^4/4! - ...
```

Range reduction maps x into [-pi, pi] using the periodicity of trigonometric functions.
Tangent is computed as `sin(x) / cos(x)`, combining the Taylor approximations of both.

```python
angle: Float = 0.7854  # approximately pi/4
s = math.sin(angle)    # approx 0.7071
c = math.cos(angle)    # approx 0.7071
t = math.tan(angle)    # approx 1.0
```

### Hyperbolic Functions: sinh(x), cosh(x), tanh(x)

The hyperbolic functions use Taylor series analogous to their trigonometric counterparts:

```
sinh(x) = x + x^3/3! + x^5/5! + ...
cosh(x) = 1 + x^2/2! + x^4/4! + ...
tanh(x) = sinh(x) / cosh(x)
```

Range reduction for sinh and cosh follows the same strategy as exp(x).

```python
x: Float = 1.0
s = math.sinh(x)  # approx 1.1752
c = math.cosh(x)  # approx 1.5431
t = math.tanh(x)  # approx 0.7616
```

### Square Root: sqrt(x)

Square root is approximated via Newton's method or a Taylor series expansion of
`sqrt(1 + u)` after writing `x = a^2 * (1 + u)` where a^2 is the nearest perfect power
of two.

```
sqrt(1 + u) = 1 + u/2 - u^2/8 + u^3/16 - ...
```

```python
x: Float = 2.0
y = math.sqrt(x)  # approx 1.4142
```

### Absolute Value: fabs(x)

Absolute value does not require a Taylor series. It is implemented using the sign bit:

```
fabs(x) = merge(x >= 0, x, -x)
```

This requires a comparison (sign check via range proof) and a conditional negation,
totaling 2--3 constraints.

```python
x: Float = -3.14
y = math.fabs(x)  # y = 3.14
```

### Inverse: inv(x)

The multiplicative inverse `1/x` is computed by finding the field element y such that
`x * y = 2^s` (the fixed-point representation of 1.0). The compiler introduces y as a
witness and constrains `x * y = 2^s` directly.

```python
x: Float = 4.0
y = 1.0 / x  # y = 0.25
```

This costs approximately 5--10 constraints including the range proof that the result is
within bounds.

## Precision Guarantees

### Approximation Error Bound

For every supported function, the compiler guarantees:

```
|f_approx(x) - f_true(x)| < 2^(-s) = 2^(-64) approx 5.42 * 10^(-20)
```

within the valid input range. This means the approximation error is below the fixed-point
resolution -- the encoded field element for the approximation is at most 1 ULP away from
the encoded field element for the true value.

### Composition of Functions

When non-linear functions are composed (e.g., `exp(sin(x))`), each function introduces
up to 1 ULP of error. For a chain of k function applications, the total error is bounded
by approximately:

```
k * 2^(-64) + higher-order terms
```

For practical chains (k &lt; 100), this remains well below FP64 precision.

### Input Range Limitations

Some functions have restricted domains:

- `math.log(x)` requires x &gt; 0. The compiler inserts a constraint that x is positive.
- `math.sqrt(x)` requires x &gt;= 0.
- `math.tan(x)` is undefined at x = pi/2 + n*pi. The compiler does not check for this --
  using tan near its poles produces unpredictable results.
- `1.0 / x` requires x != 0. The compiler constrains x to be non-zero.

Violating these domain constraints will cause proof generation to fail.

## Practical Considerations

1. **Circuit size scales with degree.** Each non-linear function call adds 30--80
   constraints to the circuit. If you call `math.exp` in a loop of 1000 iterations, expect
   on the order of 40,000--50,000 additional constraints from that function alone. Plan
   your circuit budget accordingly.

2. **Prefer simpler approximations when possible.** If you need an approximate sigmoid, a
   low-degree polynomial approximation written directly in Zinnia may be cheaper than
   composing `exp` and `div`:

   ```python
   # Expensive: uses exp + division
   def sigmoid_exact(x: Float) -> Float:
       return 1.0 / (1.0 + math.exp(-x))

   # Cheaper: degree-3 polynomial approximation (valid for |x| < 5)
   def sigmoid_approx(x: Float) -> Float:
       return 0.5 + 0.25 * x - 0.0208 * x * x * x
   ```

3. **Range reduction adds cost.** Functions applied to values far from the Taylor expansion
   point incur extra constraints for range reduction. Normalizing your data to a narrow
   range before applying non-linear functions can reduce circuit size.

4. **Use fabs and inv directly.** These functions are much cheaper than the transcendental
   functions because they do not require Taylor series. Prefer them over more expensive
   alternatives when algebraically equivalent.

5. **Batch evaluations share no state.** Each call to a non-linear function is independent.
   Evaluating `math.sin(x)` on each element of a 100-element array produces 100
   independent polynomial evaluations. There is no amortization across calls.
